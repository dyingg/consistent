# Inbox Goal — Design

**Date:** 2026-04-22
**Status:** Approved

## Problem

Every task in the codebase must belong to a goal (`tasks.goal_id NOT NULL`). This is correct for anything part of an ongoing objective, but leaves no home for genuine one-offs — "pick up dry cleaning Friday," "email the landlord." Users today either create throwaway goals (polluting the goals list) or skip tracking one-offs altogether.

## Solution

Give every user a singleton **Inbox** goal. It's a real `goals` row, marked with a dedicated column, auto-created on signup, non-deletable, and default destination for tasks the coach judges to be one-offs.

## Design

### 1. Schema marker — `isInbox` boolean

Add `is_inbox BOOLEAN NOT NULL DEFAULT false` to `goals`. Enforce "at most one Inbox per user" at the DB level via a partial unique index:

```sql
CREATE UNIQUE INDEX idx_goals_one_inbox_per_user
  ON goals(user_id) WHERE is_inbox = true;
```

A column beats an enum here — no other system-goal types are planned, and migrating column → enum later is mechanical.

### 2. Creation on signup — Better Auth hook

Wire a `databaseHooks.user.create.after` in `packages/auth/src/auth.ts`:

```ts
databaseHooks: {
  user: {
    create: {
      after: async (user) => {
        await db.insert(goals).values({
          userId: user.id,
          title: "Inbox",
          isInbox: true,
        });
      },
    },
  },
}
```

Hook failure = signup failure. No user without an Inbox. Application-layer: keeps the auth table clean of domain triggers, matches how the rest of the domain attaches to user events.

### 3. Backfill — idempotent SQL migration

New `db:generate:custom --name=backfill_inbox_goals`:

```sql
INSERT INTO goals (user_id, title, is_inbox)
SELECT u.id, 'Inbox', true
FROM "user" u
WHERE NOT EXISTS (
  SELECT 1 FROM goals g
  WHERE g.user_id = u.id AND g.is_inbox = true
);
```

Runs on next `pnpm db:migrate`. Idempotent. Handles the current single user plus any other pre-hook accounts.

### 4. Delete protection — defense in depth

**DB trigger** (custom migration):

```sql
CREATE OR REPLACE FUNCTION prevent_inbox_delete() RETURNS trigger
LANGUAGE plpgsql AS $$
BEGIN
  IF OLD.is_inbox = true
     AND EXISTS (SELECT 1 FROM "user" WHERE id = OLD.user_id) THEN
    RAISE EXCEPTION 'Cannot delete the Inbox goal'
      USING ERRCODE = 'check_violation';
  END IF;
  RETURN OLD;
END;
$$;

CREATE TRIGGER prevent_inbox_delete_trigger
  BEFORE DELETE ON goals
  FOR EACH ROW EXECUTE FUNCTION prevent_inbox_delete();
```

The `EXISTS` guard lets the cascade from `user` deletion proceed (when a user is deleted, their Inbox can go with them). Direct deletes of an Inbox while its owner still exists are rejected.

**Service guard** — `GoalsService.delete()` reads the goal (already does for ownership), checks `isInbox`, throws `ForbiddenException("The Inbox goal cannot be deleted")` before touching the repo. Clean 403 for REST + agent callers.

**Tool description** — `delete-goal` tool adds "The Inbox goal cannot be deleted" to its description. Safety rail against the model even attempting.

### 5. Agent integration — server-side Inbox resolution

**No prompt injection. No dynamic `instructions`.** The `COACH_SYSTEM_PROMPT` stays byte-stable so Anthropic's prompt cache stays globally warm.

- `create-task` tool: `goalId` becomes optional. When omitted, the tool calls `goalsService.findInboxId(userId)` and uses the resolved id.
- `bulk-create-tasks` tool: same treatment.
- Zod description on `goalId`: *"Omit when the task is a one-off not tied to any goal — it will be placed in the user's Inbox."*
- Coach prompt (`prompts/system.ts`) adds guidance:
  > You can omit `goalId` on `create-task`/`bulk-create-tasks` to drop a task into the user's Inbox. Use this for genuine one-offs. If a task looks like part of a multi-step objective — recurring, multi-sprint, tied to an ongoing ambition — pause and suggest creating a goal first. When you omit `goalId`, briefly mention the task is going to the Inbox.

### 6. Inbox is renameable, not deletable

The marker is the column, not the title. Users can rename "Inbox" to "Quick tasks" or anything else via `update-goal`. Status edits and every other field remain normal. Deletion is the only operation blocked.

## Tests

- `GoalsService.delete()` rejects with `ForbiddenException` when target is Inbox
- `GoalsRepository.findInboxByUserId` returns the Inbox row; returns null when missing
- Repository: inserting a second Inbox for the same user fails on the unique index
- Task tools: `create-task` without `goalId` resolves to Inbox; passing an explicit `goalId` bypasses Inbox resolution
- Manual: new signup via Better Auth → Inbox row present

## Out of scope

- UI treatment (pinning Inbox, distinct icon, etc.) — dashboard already renders all active goals; can revisit
- Making Inbox non-renameable — unnecessary; column is the source of truth
- Separate `create-inbox-task` tool — redundant once `goalId` is optional
- `tasks.goal_id` nullability — rejected in favor of the singleton-goal design

## Open micro-decisions — resolved

- **Cascade-from-user delete:** trigger allows it via `EXISTS` guard on owning user.
- **Renameable:** yes.
- **Routing mechanism:** tool-side lookup (not prompt injection).

## Tradeoffs accepted

- One extra DB lookup per task creation when `goalId` is omitted. Negligible (partial-index-backed, single row).
- Inbox appears in `get-goals` results alongside regular goals, flagged with `isInbox: true`. The coach sees it; the prompt trusts it to judge when "show all your goals" should include or exclude Inbox.
